## 1. 코딩테스트 문제를 읽는 순서

코딩테스트는 무작정 코드를 치면 거의 항상 틀린다.  
문제를 읽을 때는 아래 순서로 판단한다.

### 1. 입력 크기 확인

가장 먼저 봐야 하는 것은 `N`, `M`, 문자열 길이, 좌표 범위, 값의 범위다.

- `N <= 20`
완전탐색, 부분집합, 조합, 순열, DFS를 의심한다.
- `N <= 1000`
`O(N^2)`까지 가능할 때가 많다.
- `N <= 100000`
`O(N^2)`는 대부분 불가능하다. `O(N)`, `O(N log N)`을 먼저 생각한다.
- `정답의 범위가 매우 큼`
숫자 자체를 직접 찾는 것이 아니라 이분탐색으로 "가능한가/불가능한가"를 판별하는 문제일 수 있다.

### 2. 무엇을 구하라는 문제인지 확인

- 개수
경우의 수, 빈도, 조합, DP 가능성이 높다.
- 최대/최소
그리디, DP, 이분탐색, 정렬을 의심한다.
- 가능/불가능
DFS, BFS, 백트래킹, 유니온 파인드가 자주 나온다.
- 최소 횟수, 최단 거리
BFS가 매우 강한 후보이다.
- 모든 경우 출력
DFS, 백트래킹일 확률이 높다.

### 3. 자료의 모양 확인

- 문자열
문자 비교, 인덱스 처리, 정제, 투 포인터
- 1차원 배열
정렬, 누적합, 투 포인터, 슬라이딩 윈도우, 해시
- 2차원 배열
구현, 시뮬레이션, BFS, DFS
- 그래프
DFS, BFS, Union-Find, 최단거리, 위상정렬

### 4. 완전탐색이 가능한지 먼저 본다

초보자는 자꾸 고급 알고리즘부터 떠올리려 한다.  
하지만 코딩테스트는 먼저 "그냥 다 해봐도 되나?"를 확인하는 것이 더 중요하다.

예를 들어 `N = 10`이면:

- 부분집합 개수는 `2^10 = 1024`
- 순열 개수는 `10!`이라 매우 크다

즉, 같은 완전탐색이어도 무엇을 전부 시도하는지에 따라 가능 여부가 달라진다.

### 5. 시간 복잡도를 말로 설명할 수 있어야 한다

코드를 짜기 전에 아래 문장이 입으로 나와야 한다.

- "배열을 한 번만 순회하니 `O(N)`이다."
- "정렬이 들어가니 `O(N log N)`이다."
- "모든 쌍을 비교하니 `O(N^2)`다."
- "상태가 `N * M`개이고 각 상태를 한 번씩만 방문하니 `O(NM)`이다."

이 설명이 안 되면 풀이도 흔들린다.

## 2. 코딩테스트에서 자주 쓰는 판단 기준

### 완전탐색을 먼저 의심해야 하는 표현

- 모든 경우
- 가능한 수열
- 부분집합
- 조합
- 순열
- 경우의 수

### BFS를 먼저 의심해야 하는 표현

- 최소 횟수
- 최단 거리
- 몇 번 만에
- 며칠 후
- 가장 빠르게

### 투 포인터 또는 슬라이딩 윈도우를 의심해야 하는 표현

- 연속 부분수열
- 구간의 합
- 길이가 K인 구간
- 최대 길이
- 두 배열
- 정렬된 배열에서 두 수

### 해시를 의심해야 하는 표현

- 빈도
- 개수 세기
- 중복 제거
- 같은 원소인지 비교
- 아나그램

### DP를 의심해야 하는 표현

- 방법의 수
- 최대 점수
- 최소 비용
- 이전 결과를 재사용
- 같은 계산이 반복

## 3. 시간복잡도 아주 쉽게 이해하기

시간복잡도는 "입력이 커질수록 연산량이 얼마나 늘어나는가"를 보는 기준이다.  
정확한 실행 시간을 초 단위로 계산하는 개념이 아니라, 대략 얼마나 빨리 느려지는지를 보는 개념이라고 생각하면 된다.

### 왜 필요한가

코딩테스트는 정답만 맞추면 끝이 아니다.  
제한 시간 안에 돌아가야 맞는 답이다.

예를 들어 숫자 100개를 처리하는 코드는 빨라도,  
숫자 100000개가 들어오면 갑자기 너무 느려질 수 있다.  
그래서 "이 코드가 입력이 커져도 버틸까?"를 판단해야 한다.

### 가장 쉬운 비유

학생 이름을 찾는다고 하자.

- 맨 앞부터 한 명씩 찾는다.
- 정렬된 명단에서 가운데부터 찾는다.

두 방법 다 찾을 수는 있지만,  
사람 수가 많아질수록 속도 차이가 크게 난다.

이 차이를 설명하는 언어가 시간복잡도다.

### 자주 보는 시간복잡도

#### `O(1)`

입력 크기와 상관없이 거의 같은 횟수만 처리한다.

예:

- 배열에서 `arr[5]` 꺼내기
- 스택 top 보기
- 해시맵에서 키 조회 평균

직관:

사람이 10명이든 100만명이든,  
이미 위치를 알고 있으면 바로 꺼내는 느낌이다.

#### `O(N)`

입력 크기만큼 한 번 훑는다.

예:

- 배열 합 구하기
- 최대값 찾기
- 문자열 한 번 순회

직관:

학생 명단을 앞에서 끝까지 한 번 보는 느낌이다.

#### `O(N^2)`

입력을 두 겹 반복해서 본다.

예:

- 모든 쌍 비교
- 이중 반복문
- 버블 정렬

직관:

학생 한 명마다 다시 전체 학생을 전부 비교하는 느낌이다.

`N = 1000`이면 `1000000` 수준이라 아직 가능할 때가 있지만,  
`N = 100000`이면 거의 불가능하다.

#### `O(log N)`

볼 범위를 절반씩 줄인다.

예:

- 이분탐색

직관:

책 한 권에서 페이지를 찾을 때 매번 반으로 줄여 가는 느낌이다.

#### `O(N log N)`

한 번 훑는 일과 반씩 줄이는 일이 섞인 경우가 많다.

예:

- 정렬
- 정렬 후 탐색

코딩테스트에서 매우 자주 보는 안전한 상한선이다.

### 숫자로 감각 잡기

정확한 기준은 문제마다 다르지만, 초보자는 아래 정도로 감각을 잡으면 된다.

- `N = 100`이면 `O(N^2)`도 대체로 가능
- `N = 1000`이면 `O(N^2)`는 상황에 따라 가능
- `N = 100000`이면 `O(N^2)`는 거의 불가능
- `N = 100000`에서 보통 `O(N)` 또는 `O(N log N)`을 먼저 생각

### 왜 `O(N^2)`가 위험한가

예를 들어 `N = 100000`이라고 하자.

- `O(N)`은 대략 100000번 수준
- `O(N^2)`는 대략 100억 번 수준

이 차이는 매우 크다.  
그래서 배열 두 번 중첩 순회를 보기만 해도 "이 입력 크기에서 괜찮나?"를 바로 의심해야 한다.

### 이 문제집에서 자주 만나는 시간복잡도 예시

#### 문자열 한 번 순회

```java
for (int i = 0; i < s.length(); i++) {
    // 작업
}
```

시간복잡도:

- `O(N)`

#### 이중 반복문

```java
for (int i = 0; i < n; i++) {
    for (int j = 0; j < n; j++) {
        // 작업
    }
}
```

시간복잡도:

- `O(N^2)`

#### 정렬

```java
Arrays.sort(arr);
```

시간복잡도:

- 보통 `O(N log N)` 감각으로 생각

#### BFS, DFS

그래프나 격자에서 각 상태를 한 번씩만 방문하면 보통:

- `O(정점 수 + 간선 수)`
- 격자판이면 `O(NM)`

쉽게 말하면:

"갈 수 있는 칸이나 노드를 한 번씩만 본다"는 뜻이다.

#### DP

`dp[i]`를 1번부터 N번까지 채우면 보통 `O(N)`이다.  
하지만 각 `i`마다 이전 값을 전부 보면 `O(N^2)`가 될 수 있다.

예:

- 피보나치 DP: `O(N)`
- LIS 기본형: `O(N^2)`

### 문제를 볼 때 시간복잡도 판단하는 순서

1. 입력 크기 `N`을 본다.
2. 내가 생각한 코드에서 반복문이 몇 겹인지 본다.
3. 정렬이 들어가는지 본다.
4. 각 상태를 여러 번 다시 계산하는지 본다.
5. "이 입력에서 진짜 돌아갈까?"를 말로 확인한다.

### 초보자가 자주 하는 오해

#### 오해 1

"정답만 맞으면 된다."

아니다. 제한 시간 안에 돌아야 한다.

#### 오해 2

"반복문 두 개면 무조건 안 된다."

아니다. `N = 100`이면 `O(N^2)`도 충분히 가능하다.

#### 오해 3

"시간복잡도는 외워야 한다."

외우는 것도 필요하지만, 더 중요한 것은 반복 구조를 보고 감각적으로 판단하는 것이다.

### 실전 빠른 판단표

- 배열 한 번 순회: `O(N)`
- 배열 두 겹 순회: `O(N^2)`
- 정렬: `O(N log N)`
- 해시 조회 평균: `O(1)`
- 이분탐색: `O(log N)`
- 격자 전체 탐색: `O(NM)`

### 처음에는 이렇게 생각하면 된다

문제를 풀 때 아래 질문만 해도 훨씬 낫다.

- 이 코드는 배열을 몇 번 도는가
- 중첩 반복문이 있는가
- 정렬이 있는가
- 같은 계산을 또 하고 있는가

이 네 질문이 시간복잡도 감각의 시작이다.

### 입력 크기별 허용 복잡도 판단표

아래 표는 절대적인 법칙은 아니지만, 초보자가 문제를 읽고 첫 판단을 내릴 때 매우 유용하다.

| 입력 크기 느낌 | 먼저 의심할 복잡도 | 대체로 가능성이 낮은 복잡도 | 대표 풀이 감각 |
|---|---|---|---|
| `N <= 20` | `O(2^N)`, `O(N!)`, `O(N^2)` | 없음. 다만 `N!`도 상황에 따라 커질 수 있음 | 부분집합, 순열, 조합, DFS, 백트래킹 |
| `N <= 100` | `O(N^2)`, `O(N^3)` 일부, `O(N log N)` | `O(2^N)` | 구현, 브루트포스, 2차원 배열, 시뮬레이션 |
| `N <= 1000` | `O(N^2)`, `O(N log N)`, `O(N)` | `O(N^3)`는 대체로 위험 | 모든 쌍 비교, DP 기본형, 정렬 |
| `N <= 10000` | `O(N log N)`, `O(N)` | `O(N^2)`는 대체로 위험 | 정렬, 해시, 투 포인터, 이분탐색 |
| `N <= 100000` | `O(N)`, `O(N log N)` | `O(N^2)` 거의 불가 | 슬라이딩 윈도우, 해시, BFS, 정렬 |
| `N <= 1000000` | `O(N)` 수준 | `O(N log N)`도 문제에 따라 부담, `O(N^2)` 불가 | 단순 순회, 누적합, 수학, 그리디 |

표를 읽는 방법:

- 입력이 작으면 완전탐색부터 본다.
- 입력이 커지면 정렬, 해시, 투 포인터, 이분탐색으로 줄여야 한다.
- `N = 100000`에서 이중 반복문이 보이면 거의 항상 다시 생각해야 한다.

### 예시로 바로 연결해 보기

#### `N <= 20`

가능한 경우를 다 만들어도 되는 경우가 많다.

예:

- 부분집합 전부 만들기
- 순열 전부 만들기
- DFS로 모든 경우 탐색

#### `N <= 1000`

이중 반복문이 아직 가능할 때가 많다.

예:

- 모든 학생 쌍 비교
- LIS 기본형 `O(N^2)`
- 2차원 배열 계산

#### `N <= 100000`

이제는 한 번 순회하거나 정렬 후 처리하는 쪽으로 가야 한다.

예:

- 해시맵 빈도 계산
- 투 포인터
- 슬라이딩 윈도우
- `Arrays.sort()` 후 탐색

### 표를 볼 때 주의할 점

#### 1. 실제 연산량은 문제마다 다르다

예를 들어 `O(N^2)`라도 안쪽 연산이 매우 단순하면 가능할 때가 있고,  
반대로 `O(N log N)`이어도 상수 시간이 큰 로직이면 부담이 될 수 있다.

하지만 입문 단계에서는 위 표 기준으로 먼저 감을 잡으면 충분하다.

#### 2. 입력이 하나만 있는 게 아니다

문제에 따라 `N`, `M`이 같이 나온다.

예:

- 그래프는 `O(N + M)`
- 격자는 `O(NM)`

즉, 변수 하나만 보지 말고 전체 상태 수를 같이 봐야 한다.

#### 3. 완전탐색도 종류에 따라 다르다

`N <= 20`이라고 해서 무조건 다 되는 것은 아니다.

- 부분집합은 `2^20` 정도라 가능성이 있다.
- 순열 `20!`은 절대 불가능하다.

그래서 "완전탐색"이라는 말만 보고 안심하면 안 되고,  
정확히 몇 경우가 나오는지 생각해야 한다.

### 초보자용 한 줄 요약

- `N`이 작으면 완전탐색
- `N`이 크면 정렬, 해시, 투 포인터, 이분탐색
- 최소 횟수면 BFS
- 같은 계산 반복이면 DP
- 이중 반복문이 보이면 먼저 입력 크기를 다시 본다

## 4. 코딩테스트를 처음 시작할 때 알아야 할 자바 기반 문법

이 문단은 문법 전체를 설명하는 문서가 아니라, 이 문제집을 읽기 전에 반드시 필요한 최소 기반을 정리한 것이다.

### 빠른 입력

자바 코딩테스트에서는 보통 아래 조합을 가장 많이 쓴다.

```java
import java.io.BufferedReader;
import java.io.InputStreamReader;
import java.io.IOException;
import java.util.StringTokenizer;
```

이유:

- `Scanner`보다 빠르다.
- 한 줄을 읽고 공백 기준으로 잘라 쓰기 편하다.

예시:

```java
BufferedReader br = new BufferedReader(new InputStreamReader(System.in));
StringTokenizer st = new StringTokenizer(br.readLine());
int n = Integer.parseInt(st.nextToken());
int m = Integer.parseInt(st.nextToken());
```

### 배열

```java
int[] arr = new int[n];
int[][] board = new int[n][m];
```

배열은 코딩테스트의 기본 저장소다.  
문자열 문제를 빼면 거의 모든 챕터에서 배열 감각이 필요하다.

### 반복문

배열과 문자열은 거의 항상 반복문으로 순회한다.

```java
for (int i = 0; i < n; i++) {
    System.out.println(arr[i]);
}
```

### 조건문

문제의 규칙을 코드로 옮길 때 가장 많이 쓴다.

```java
if (arr[i] > arr[i - 1]) {
    answer++;
}
```

### 문자열 기본

```java
String s = br.readLine();
char c = s.charAt(0);
int len = s.length();
String sub = s.substring(1, 3);
```

문자열 문제에서는 `charAt`, `length`, `substring`, `toCharArray`, `indexOf`를 자주 쓴다.

### 컬렉션 기본

```java
import java.util.HashMap;
import java.util.ArrayList;
import java.util.Stack;
import java.util.Queue;
import java.util.LinkedList;
import java.util.Deque;
import java.util.ArrayDeque;
import java.util.TreeSet;
```

문제에 따라 자료구조를 바꿔야 한다.

- 빠른 빈도 계산: `HashMap`
- 순서대로 저장: `ArrayList`
- 뒤에서 넣고 빼기: `Stack` 또는 `Deque`
- 먼저 들어온 순서대로 처리: `Queue`
- 정렬 + 중복 제거: `TreeSet`

## 5. 챕터별 학습

이 문제집은 챕터를 순서대로 풀면 자연스럽게 사고 범위가 넓어지도록 구성되어 있다.

### chap1. 문자열

#### 무엇을 배우는 챕터인가

문자열을 문자 단위로 읽고, 비교하고, 변형하는 훈련이다.  
알고리즘이라기보다 "문제를 조건대로 정확히 구현하는 힘"을 기른다.

#### 대표 문제 유형

- 특정 문자 개수 세기
- 대소문자 변환
- 단어 뒤집기
- 회문 문자열 판별
- 유효한 팰린드롬
- 숫자만 추출
- 문자열 압축

#### 핵심 사고

- 문자열은 결국 문자들의 나열이다.
- 앞에서 한 글자씩 볼 것인지, 양 끝에서 동시에 볼 것인지 결정해야 한다.
- 비교 전에 전처리하면 쉬워지는지 항상 확인한다.

예:

`"found7, time: study; Yduts; emit, 7Dnuof"` 같은 문자열에서 회문을 판별하라고 하면  
문장부호와 공백을 제거하고, 대소문자를 통일한 뒤 비교하면 훨씬 단순해진다.

#### 자주 쓰는 알고리즘적 생각

- 한 번 순회하며 조건 체크
- 양 끝 비교를 위한 투 포인터
- 문자열 정제 후 비교

#### 이 챕터에서 필요한 자바 문법

```java
String s = "abc";
char x = s.charAt(1);      // 'b'
int len = s.length();      // 3
String sub = s.substring(1, 3); // "bc"
char[] arr = s.toCharArray();
```

추가로 자주 쓰는 메서드:

- `Character.toLowerCase(c)`
- `Character.toUpperCase(c)`
- `Character.isAlphabetic(c)`
- `Character.isDigit(c)`
- `StringBuilder.reverse()`

예시:

```java
String s = "gooG";
s = s.toLowerCase();   // "goog"
```

#### 문제를 풀 때 체크할 질문

- 대소문자를 통일해야 하나
- 공백, 특수문자, 숫자를 제거해야 하나
- 양 끝 비교가 더 빠른가
- 문자열을 바로 바꾸기보다 문자 배열로 보는 편이 쉬운가

#### 자주 틀리는 부분

- `substring(begin, end)`에서 `end`가 포함된다고 착각
- 마지막 묶음 처리 누락
- 문자와 문자열 비교를 혼동

### chap2. 배열과 구현

#### 무엇을 배우는 챕터인가

문제 조건을 그대로 코드로 옮기는 연습이다.  
알고리즘 자체보다 배열 인덱스 처리, 2차원 순회, 조건 구현 정확도가 중요하다.

#### 대표 문제 유형

- 앞 수보다 큰 수 출력
- 보이는 학생 수 세기
- 피보나치 수열
- 소수 판별 및 소수 개수
- 뒤집은 소수
- 격자판 최대합
- 봉우리
- 가위바위보
- 점수 계산
- 임시반장 정하기
- 멘토링

#### 핵심 사고

- 현재 값과 이전 값을 비교한다.
- 2차원 배열은 행, 열, 대각선을 따로 본다.
- 입력 크기가 작으면 모든 경우를 직접 비교해도 된다.

예:

`7 3 9 5 6 12`에서 앞 수보다 큰 수를 찾을 때는  
현재 값이 직전 값보다 큰지만 보면 된다.

#### 알고리즘: 에라토스테네스의 체

1부터 N까지의 소수를 빠르게 구하는 대표 방법이다.

핵심 아이디어:

- 소수는 1과 자기 자신만 약수로 가진다.
- 어떤 수의 배수는 소수가 아니다.
- 그래서 2, 3, 5 같은 수의 배수를 지워 나가면 소수만 남는다.

예: `20` 이하 소수 구하기

1. 2를 본다. 2는 소수다.
2. 2의 배수인 4, 6, 8, 10, ... 을 지운다.
3. 다음으로 남아 있는 3을 본다. 3은 소수다.
4. 3의 배수인 6, 9, 12, 15, 18을 지운다.
5. 다음 남은 수 5를 본다.

최종적으로 남는 소수:

- 2
- 3
- 5
- 7
- 11
- 13
- 17
- 19

#### 이 챕터에서 필요한 자바 문법

```java
int[] arr = new int[n];
for (int i = 0; i < n; i++) {
    arr[i] = Integer.parseInt(br.readLine());
}
```

2차원 배열:

```java
int[][] board = new int[n][n];
for (int i = 0; i < n; i++) {
    StringTokenizer st = new StringTokenizer(br.readLine());
    for (int j = 0; j < n; j++) {
        board[i][j] = Integer.parseInt(st.nextToken());
    }
}
```

#### 문제를 풀 때 체크할 질문

- 인덱스 범위를 벗어나지 않는가
- 행, 열, 대각선을 따로 계산해야 하는가
- 모든 학생 쌍, 모든 날짜 쌍처럼 전부 비교해도 되는가

#### 자주 틀리는 부분

- 첫 번째 값을 예외 처리하지 않음
- 2차원 배열 경계 체크 누락
- 소수 판별에서 1을 소수로 처리

### chap3. Two Pointers / Sliding Window

#### 무엇을 배우는 챕터인가

배열이나 수열을 `O(N^2)` 대신 `O(N)` 또는 `O(N log N)` 수준으로 줄이는 사고를 배운다.

#### Two Pointers

포인터 두 개를 움직이며 문제를 해결하는 방법이다.

주로 쓰는 경우:

- 정렬된 두 배열 합치기
- 공통 원소 찾기
- 두 수의 합
- 구간 조건 관리

예:

- A: `1 3 5`
- B: `2 3 6`

현재 가리키는 두 값 중 더 작은 값을 결과에 넣고, 그쪽 포인터를 한 칸 이동한다.

진행:

1. `1`과 `2` 비교 -> `1` 추가
2. `3`과 `2` 비교 -> `2` 추가
3. `3`과 `3` 비교 -> 둘 다 추가 후 둘 다 이동

#### Sliding Window

연속된 구간을 효율적으로 관리하는 방법이다.

주로 쓰는 경우:

- 길이가 K인 연속 구간의 합
- 연속 부분수열
- 조건을 만족하는 최대 길이

예:

매출 배열이 `12 15 11 20 25 10 20 19`, `K = 3`이라고 하자.

첫 구간:

- `12 + 15 + 11 = 38`

다음 구간:

- 이전 합 `38`
- 맨 앞 `12`를 빼고
- 새 값 `20`을 더하면
- `46`

이렇게 하면 매번 세 수를 다시 더하지 않아도 된다.

#### 이 챕터에서 필요한 자바 문법

- 정렬: `Arrays.sort(arr)`
- 합 관리용 변수
- 두 개의 인덱스 `lt`, `rt`

```java
import java.util.Arrays;
Arrays.sort(arr);
```

#### 문제를 풀 때 체크할 질문

- 자료가 정렬되어 있는가
- 정렬하면 쉬워지는가
- "연속"이라는 단어가 있는가
- 이전 구간 결과를 재사용할 수 있는가

#### 자주 틀리는 부분

- 포인터 이동 조건이 꼬임
- 윈도우 시작 합을 잘못 잡음
- 정렬이 필요한데 빼먹음

### chap4. HashMap / TreeSet

#### 무엇을 배우는 챕터인가

빠른 조회, 빈도 계산, 중복 제거, 정렬된 집합 처리를 배운다.

#### HashMap

키와 값을 저장하는 자료구조다.  
코딩테스트에서는 "무엇이 몇 번 나왔는가"를 셀 때 가장 자주 쓴다.

예:

문자열 `"BACBACCACCBDEDE"`에서 빈도를 세면:

- A: 4
- B: 3
- C: 5
- D: 2
- E: 2

이런 문제를 배열로도 풀 수 있지만, 문자의 종류가 많거나 문자열이 길면 `HashMap`이 훨씬 편하다.

#### TreeSet

중복을 제거하면서 자동 정렬까지 해주는 자료구조다.

주로 쓰는 경우:

- 중복 없는 수만 관리
- 큰 수부터, 작은 수부터 확인
- K번째 큰 수

#### Sliding Window + HashMap

이 챕터는 해시만 배우는 것이 아니라 "구간 안의 빈도"도 자주 다룬다.

예:

매출액의 종류 문제에서 길이 `K` 구간마다 서로 다른 숫자 개수를 구할 수 있다.  
왼쪽 값을 하나 빼고, 오른쪽 값을 하나 넣으면서 `HashMap` 빈도를 갱신한다.

#### 이 챕터에서 필요한 자바 문법

```java
import java.util.HashMap;
import java.util.TreeSet;
```

빈도 계산:

```java
map.put(x, map.getOrDefault(x, 0) + 1);
```

빈도가 0이 되면 삭제:

```java
map.put(x, map.get(x) - 1);
if (map.get(x) == 0) {
    map.remove(x);
}
```

#### 문제를 풀 때 체크할 질문

- 특정 값이 몇 번 등장했는가
- 같은 구성을 가진 문자열인가
- 중복 제거와 정렬이 동시에 필요한가
- 현재 구간의 상태를 빈도로 관리할 수 있는가

#### 자주 틀리는 부분

- 빈도 0인 키를 삭제하지 않음
- `HashMap`의 순서를 믿음
- 문자열 비교를 `==`로 함

### chap5. Stack / Queue

#### 무엇을 배우는 챕터인가

순서 제어가 중요한 문제를 푸는 방법을 배운다.

#### Stack

나중에 들어온 것이 먼저 나온다.  
괄호, 수식, 되돌아가기 구조에 강하다.

주요 문제:

- 올바른 괄호
- 괄호 안 문자 제거
- 후위식 계산
- 쇠막대기

괄호 문제 예:

1. `(`를 만나면 스택에 넣는다.
2. `)`를 만나면 스택에서 하나 꺼낸다.
3. 꺼낼 것이 없는데 `)`가 나오면 잘못된 문자열이다.
4. 끝났을 때 스택이 비어 있어야 정상이다.

#### Queue

먼저 들어온 것이 먼저 나온다.

주요 문제:

- 공주 구하기
- 응급실
- BFS의 기본 자료구조

응급실 문제 예:

1. 맨 앞 환자를 꺼낸다.
2. 더 높은 위험도가 뒤에 있으면 다시 맨 뒤로 보낸다.
3. 아니면 진료한다.

#### 이 챕터에서 필요한 자바 문법

실전에서는 `Stack`보다 `Deque`를 더 많이 권장한다.

```java
import java.util.Deque;
import java.util.ArrayDeque;
import java.util.Queue;
import java.util.LinkedList;
```

스택처럼 사용:

```java
Deque<Character> stack = new ArrayDeque<>();
stack.push('(');
char x = stack.pop();
char top = stack.peek();
```

큐처럼 사용:

```java
Queue<Integer> queue = new LinkedList<>();
queue.offer(1);
int x = queue.poll();
int y = queue.peek();
```

#### 문제를 풀 때 체크할 질문

- 최근에 들어온 것을 먼저 꺼내야 하는가
- 먼저 들어온 순서대로 처리해야 하는가
- 중간 삽입보다 앞뒤 제거가 중요한가

#### 자주 틀리는 부분

- 비어 있는 자료구조에서 꺼냄
- 괄호 짝을 끝까지 확인하지 않음
- 큐에서 다시 넣는 조건을 잘못 작성

### chap6. Sorting / Searching

#### 무엇을 배우는 챕터인가

정렬을 통해 문제를 단순화하고, 이분탐색과 결정 알고리즘으로 답을 빠르게 찾는 사고를 배운다.

#### 정렬

정렬의 힘은 "비교 기준을 통일한다"는 데 있다.

예:

- 좌표를 정렬하면 겹침 판단이 쉬워진다.
- 숫자를 정렬하면 두 수 비교가 쉬워진다.
- 회의 시간을 끝나는 시간 기준으로 정렬하면 그리디 선택이 쉬워진다.

#### 이분탐색

정렬된 자료에서 중간값을 기준으로 범위를 반씩 줄여 나간다.

예:

정렬된 배열 `1 3 5 7 9`에서 `7` 찾기

1. 가운데 `5` 확인
2. `7`은 더 크므로 오른쪽으로 이동
3. 가운데 `7` 확인
4. 찾음

시간 복잡도는 `O(log N)`이다.

#### 결정 알고리즘

답을 직접 찾는 것이 아니라  
"이 값으로 가능한가?"를 판단하는 방식이다.

대표 문제:

- 마구간 정하기
- 뮤직비디오
- 랜선 자르기 유형

예:

"DVD 용량이 17이면 모든 곡을 M장 안에 담을 수 있는가?"

이 질문에 대해 `가능/불가능`을 판별하는 함수를 만든 뒤,  
이분탐색으로 최소 가능한 용량을 찾는다.

#### 이 챕터에서 필요한 자바 문법

```java
import java.util.Arrays;
```

기본 정렬:

```java
Arrays.sort(arr);
```

객체 정렬은 `Comparable` 또는 `Comparator`가 필요하다.

```java
Arrays.sort(points, (a, b) -> a.x - b.x);
```

#### 문제를 풀 때 체크할 질문

- 정렬하면 규칙이 더 분명해지는가
- 답의 범위를 숫자로 잡을 수 있는가
- "가능하면 더 큰 값도 가능한가" 같은 단조성이 있는가

#### 자주 틀리는 부분

- 이분탐색 시작값, 끝값 설정 오류
- 판별 함수가 정확하지 않음
- 정렬 후 원래 순서를 잊고 잘못 해석

### chap7. Recursive / Tree / Graph DFS, BFS 기초

#### 무엇을 배우는 챕터인가

재귀 함수의 구조를 이해하고, 트리와 그래프 탐색의 가장 기본 형태를 익히는 챕터다.

#### 재귀

함수가 자기 자신을 다시 호출하는 방식이다.

재귀가 성립하려면 두 가지가 있어야 한다.

- 종료 조건
- 더 작은 문제로 가는 과정

예: 자연수 `N`부터 1까지 출력

```java
void dfs(int n) {
    if (n == 0) return;
    dfs(n - 1);
    System.out.print(n + " ");
}
```

#### 트리 순회

트리는 부모와 자식 관계를 가지는 구조다.  
전위, 중위, 후위 순회는 방문 순서가 다르다.

- 전위: 부모 -> 왼쪽 -> 오른쪽
- 중위: 왼쪽 -> 부모 -> 오른쪽
- 후위: 왼쪽 -> 오른쪽 -> 부모

#### 이 챕터에서 필요한 자바 문법

- 재귀 메서드 선언
- 전역 배열, 전역 변수
- `ArrayList<Integer>[]` 같은 인접 리스트

```java
import java.util.ArrayList;
```

#### 자주 틀리는 부분

- 종료 조건 없이 재귀 호출
- 방문 체크 누락으로 무한 순환
- 재귀 흐름을 머릿속으로만 추적하려고 함

### chap8. DFS / BFS 활용

#### 무엇을 배우는 챕터인가

이 챕터부터는 탐색을 본격적으로 문제 해결 도구로 사용한다.

#### DFS가 잘 맞는 문제

- 부분집합
- 조합
- 순열
- 중복순열
- 미로 경로 수
- 바둑이 승차
- 수열 추측하기

DFS의 핵심은 "선택지를 만들고 끝까지 내려간다"는 것이다.

예:

숫자 1, 2, 3 중 몇 개를 골라 부분집합을 만든다면  
각 숫자마다 선택은 두 개다.

- 넣는다
- 넣지 않는다

이 선택을 끝까지 내려가며 모든 경우를 만든다.

#### BFS가 잘 맞는 문제

- 최단 거리
- 최소 이동 횟수
- 섬의 개수는 DFS/BFS 둘 다 가능
- 미로 최단거리

예:

5에서 시작해서 `-1`, `+1`, `+5` 이동이 가능할 때  
14까지 가는 최소 횟수는 BFS로 찾는다.

이유:

가까운 상태부터 탐색하므로 처음 도착하는 순간이 최소 횟수다.

#### 이 챕터에서 필요한 자바 문법

- 재귀
- 큐
- `dx`, `dy` 방향 배열
- 방문 배열

```java
int[] dx = {-1, 0, 1, 0};
int[] dy = {0, 1, 0, -1};
```

#### 문제를 풀 때 체크할 질문

- 가능한 경우를 전부 탐색해야 하나
- 최소 횟수를 구해야 하나
- 방문 체크가 필요한 상태 공간인가

#### 자주 틀리는 부분

- DFS에서 복구 누락
- BFS에서 큐에 넣을 때 방문 체크 안 함
- 좌표 범위 체크 누락

### chap9. Greedy

#### 무엇을 배우는 챕터인가

매 순간 가장 좋아 보이는 선택을 하는 전략을 배운다.  
단, 그 선택이 전체 최적해로 이어지는 근거가 있어야 한다.

#### 그리디가 맞는 전형

- 회의실 배정
- 씨름 선수
- 원더랜드 간선 선택 일부 변형
- 가장 빨리 끝나는 것 선택

#### 대표 예: 회의실 배정

회의를 최대한 많이 넣으려면  
빨리 끝나는 회의부터 선택하는 것이 유리하다.

왜냐하면 빨리 끝나야 뒤에 더 많은 회의를 넣을 공간이 생기기 때문이다.

이 문제는 보통 끝나는 시간 기준 오름차순,  
끝나는 시간이 같으면 시작 시간 기준 오름차순 정렬 후 푼다.

#### 이 챕터에서 필요한 자바 문법

- 객체 배열 정렬
- `Comparable`, `Comparator`

```java
Arrays.sort(meetings, (a, b) -> {
    if (a.end == b.end) return a.start - b.start;
    return a.end - b.end;
});
```

#### 문제를 풀 때 체크할 질문

- 지금 최선의 선택이 이후 선택의 폭을 넓히는가
- 정렬 기준이 명확한가
- 반례를 떠올려도 이 선택이 유지되는가

#### 자주 틀리는 부분

- 근거 없이 그리디로 단정
- 정렬 기준을 잘못 잡음
- 같은 끝시간 처리 누락

### chap10. Dynamic Programming

#### 무엇을 배우는 챕터인가

이전 계산 결과를 저장해서 다시 쓰는 방법을 배운다.

#### DP를 써야 하는 상황

- 같은 계산이 여러 번 반복된다.
- 작은 문제의 답으로 큰 문제를 만들 수 있다.
- 최대값, 최소값, 경우의 수 문제가 많다.

#### 예: 계단 오르기

한 번에 1칸 또는 2칸씩 갈 수 있다고 하자.

`dp[i] = i번 계단에 도달하는 방법의 수`

그러면:

- `dp[1] = 1`
- `dp[2] = 2`
- `dp[3] = dp[2] + dp[1] = 3`

왜냐하면 3번 계단에 오는 마지막 방법은 두 가지뿐이기 때문이다.

- 2번에서 1칸 올라오기
- 1번에서 2칸 올라오기

#### DP의 핵심 절차

1. `dp[i]`가 무엇인지 정확히 정의한다.
2. 점화식을 만든다.
3. 초기값을 넣는다.
4. 계산 순서를 정한다.

#### 이 챕터에서 필요한 자바 문법

- `int[] dp`, `long[] dp`
- 2차원 DP는 `int[][] dp`
- `Arrays.fill()`로 초기화

```java
import java.util.Arrays;
Arrays.fill(dp, -1);
```

#### 문제를 풀 때 체크할 질문

- `dp[i]`의 의미를 한 문장으로 말할 수 있는가
- 이전 어떤 상태가 현재 상태에 영향을 주는가
- 초기값은 무엇인가
- 반복문 방향은 맞는가

#### 자주 틀리는 부분

- 상태 정의가 불분명함
- 초기값 누락
- 점화식은 맞는데 반복문 순서가 틀림

## 6. 교재에는 약하지만 일반 코딩테스트에서 꼭 알아야 하는 알고리즘

이 문제집만 끝내고 바로 실전에 들어가면 부족할 수 있는 영역이다.  
아래 주제들은 대기업, 플랫폼 코테, 백준, 프로그래머스에서 매우 자주 만난다.

### 누적합

#### 언제 쓰는가

- 구간 합
- 구간 평균
- 특정 범위의 합을 여러 번 물을 때

#### 아이디어

`prefix[i] = 1번부터 i번까지의 누적 합`

그러면 `l`부터 `r`까지의 합은

`prefix[r] - prefix[l - 1]`

예:

배열 `5 2 7 3 6`

누적합:

- 5
- 7
- 14
- 17
- 23

2번부터 4번까지의 합:

- `17 - 5 = 12`

#### 왜 중요한가

구간 합을 매번 다시 더하면 `O(N)`이지만,  
누적합을 쓰면 한 번의 질의가 `O(1)`이 된다.

### 이차원 누적합

#### 언제 쓰는가

- 직사각형 범위 합
- 격자판 합을 여러 번 물을 때

#### 핵심

`sum[x][y]`를 `(1,1)`부터 `(x,y)`까지의 누적합으로 만든다.

직사각형 합은 포함과 제외를 이용한다.

### 유니온 파인드

#### 언제 쓰는가

- 같은 집합인지 확인
- 연결 여부 확인
- 사이클 판별
- 최소 스패닝 트리

#### 핵심

각 원소가 어느 집합의 대표를 가지는지 관리한다.

주요 연산:

- `find(x)`: x가 속한 집합의 대표 찾기
- `union(a, b)`: 두 집합 합치기

예:

1과 2를 합치고, 2와 3을 합치면  
1과 3은 같은 집합이 된다.

### 다익스트라

#### 언제 쓰는가

- 가중치가 있는 그래프에서 최단거리
- 비용, 거리, 시간 최소화

#### 핵심 조건

- 간선 가중치가 음수가 아니어야 한다.

#### 아이디어

현재까지 가장 짧은 거리가 확정된 정점부터 처리한다.  
보통 `PriorityQueue`를 사용한다.

예:

도시 A에서 다른 도시까지 버스 비용 최소값

이런 표현이 나오면 다익스트라 가능성이 높다.

### 위상정렬

#### 언제 쓰는가

- 선수 과목
- 작업 순서
- 먼저 해야 하는 일이 있는 경우

#### 핵심

방향 그래프에서 진입차수가 0인 정점부터 처리한다.

예:

- 1번 과목을 들어야 3번 가능
- 2번 과목을 들어야 3번 가능

그러면 1, 2가 먼저 오고 3이 나중에 와야 한다.

### 최소 스패닝 트리

#### 언제 쓰는가

- 모든 도시를 최소 비용으로 연결
- 모든 컴퓨터를 최소 케이블 비용으로 연결

#### 대표 알고리즘

- 크루스칼
- 프림

입문 단계에서는 보통 크루스칼과 유니온 파인드를 먼저 배운다.

### 비트마스킹

#### 언제 쓰는가

- 부분집합 관리
- 방문 집합 표현
- 원소가 적은 상태를 압축해서 저장

#### 핵심

정수를 이진수로 보고 각 비트를 선택 여부로 사용한다.

예:

원소가 4개면:

- `0101`은 1번, 3번 원소 선택 같은 식으로 해석할 수 있다.

### 문자열 고급 알고리즘

입문 단계에서는 바로 필요하지 않을 수 있지만, 이후 자주 본다.

- KMP
문자열 검색을 빠르게 한다.
- Trie
문자열 집합, 접두사 검색에 강하다.

### 세그먼트 트리 / 펜윅 트리

#### 언제 배우면 되는가

입문 직후에는 우선순위가 가장 높지는 않다.  
하지만 구간 합, 구간 최소/최대 갱신 문제가 많아지면 필요해진다.

#### 지금 단계에서 기억할 것

- 누적합은 "값이 잘 안 바뀌는 구간 합"에 강하다.
- 세그먼트 트리는 "값이 자주 바뀌는 구간 질의"에 강하다.
